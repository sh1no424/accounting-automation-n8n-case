# Нефункциональные требования

| ID      | Категория              | Требование                                                                                                                                           |
| ------- | ---------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| NFR-001 | Производительность     | Система должна проверять наличие новых документов в СБИС с периодичностью не реже одного раза в 15 минут.                                            |
| NFR-002 | Производительность     | Система должна выполнять обработку документов асинхронно, чтобы ошибка одного документа не блокировала обработку других документов.                  |
| NFR-003 | Надёжность             | Система не должна повторно обрабатывать документ из СБИС, если документ с таким же ID уже был обработан ранее.                                       |
| NFR-004 | Надёжность             | При ошибке обработки документ не должен теряться. Система должна сохранить статус ошибки и причину сбоя.                                             |
| NFR-005 | Надёжность             | Если 1С недоступна, система не должна создавать некорректный или неполный документ поступления.                                                      |
| NFR-006 | Надёжность             | Если внешний сервис OCR/LLM недоступен или вернул некорректный ответ, документ должен быть передан на ручную проверку.                               |
| NFR-007 | Надёжность             | Если сервис проверки контрагента недоступен, причина невозможности проверки должна быть зафиксирована в результате обработки документа.              |
| NFR-008 | Безопасность           | Доступ к административному интерфейсу управления правилами поставщиков должен быть ограничен авторизованными пользователями или защищённым контуром. |
| NFR-009 | Целостность данных     | Система должна использовать единый JSON-формат для передачи данных между этапами обработки документа.                                                |
| NFR-010 | Целостность данных     | Система должна сохранять исходное наименование услуги или лизинговой позиции при подстановке обобщённой номенклатуры.                                |
| NFR-011 | Логирование            | Система должна фиксировать причины, по которым документ получил статус «Требует ручной проверки».                                                    |
| NFR-012 | Логирование            | Система должна фиксировать ошибки интеграций с внешними системами: СБИС, почтой, OCR/LLM, DaData/ФНС и 1С.                                           |
| NFR-013 | Наблюдаемость          | По результату обработки документа должен быть понятен итоговый статус и список пройденных или непройденных проверок.                                 |
| NFR-014 | Удобство сопровождения | Правила счетов учета поставщиков должны настраиваться через административный интерфейс без изменения основного workflow.                             |
| NFR-015 | Масштабируемость       | Система должна позволять добавлять новые источники документов без полной переработки основного процесса обработки.                                   |
| NFR-016 | Масштабируемость       | Система должна позволять добавлять новые проверки документа без изменения логики получения документов из источников.                                 |
| NFR-017 | Совместимость          | Система должна поддерживать обработку документов в форматах XML, PDF и изображений.                                                                  |
| NFR-018 | Совместимость          | Интеграция с 1С должна выполняться через OData API.                                                                                                  |
| NFR-019 | Удобство пользователя  | Уведомление бухгалтеру должно содержать понятное описание проблемы, а не только технический код ошибки.                                              |
| NFR-020 | Удобство пользователя  | Административный интерфейс правил поставщиков должен позволять создавать, изменять, включать и отключать правила без работы с кодом.                 |
| NFR-021 | Тестируемость          | Для основных сценариев обработки должны быть подготовлены тестовые примеры входных данных, успешного результата и ошибки обработки.                  |
